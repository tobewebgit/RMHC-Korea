# Git 커밋 규칙 및 가이드라인 (Git Commit Guidelines)

본 문서는 RMHC Korea 홈페이지 개편 퍼블리싱 프로젝트의 형상 관리와 소규모 단위 작업을 안전하게 추적 및 기록하기 위한 커밋 컨벤션을 정의합니다.

---

## 1. 커밋 메시지 기본 구조 (Commit Message Format)

커밋 메시지는 **Conventional Commits** 규격을 기반으로 작성하며, 한글로 설명합니다.

```text
<type>(<scope>): <subject>

[Optional Body]
```

### 1) Type (유형)
- **`feat`**: 새로운 페이지 퍼블리싱, 신규 기능 또는 HTML 공통 컴포넌트 추가
- **`style`**: CSS 수정, 글로벌 디자인 토큰 재설정, 폰트/색상 변경 등 (로직 변화 없음)
- **`refactor`**: 폴더 구조 재배치, HTML 인클루드 경로 정리, 중복 마크업 개선 등
- **`docs`**: 업무일지 작성, 가이드 문서 추가/수정
- **`chore`**: 패키지 설치, 빌드 설정(`vite.config.js`), `.gitignore` 수정 등
- **`fix`**: 마크업 깨짐 수정, 빌드 경고 해결, 오타 수정 등

### 2) Scope (범위 - 선택사항)
작업이 적용된 구역을 명시합니다. (예: `about`, `donate`, `design-system`, `build` 등)

---

## 2. 소규모 커밋 전략 (Micro-Commit Strategy)

1. **논리적 작업 단위 커밋 (적정 단위 묶음)**:
   - 한 번에 대량의 변경 사항을 한꺼번에 커밋하지 않고, 의미 있는 작업 단위로 커밋합니다.
   - 단, 너무 미세하고 작은 변경 단위(예: 사소한 오타 수정, 개별 주석 추가 등)는 단독으로 즉시 커밋하지 않습니다. 이러한 작은 작업 단위가 최소 5개 이상 누적되었을 때 비로소 하나의 커밋으로 묶어서 진행합니다.
2. **커밋 전 빌드 확인**:
   - 커밋하기 전에 항상 `npm run build`를 실행하여 컴파일 오류나 누락된 종속성이 없는지 사전 검증합니다.
3. **일 단위 업무일지 연동**:
   - 매일 작업이 끝나면 일 단위 업무일지를 작성하고, 해당 일지를 리포지토리에 커밋하여 관리합니다.

---

## 3. 예시 (Examples)

- `style(design-system): 웜 라이트 테마 디자인 토큰 및 style.css 설정`
- `refactor(includes): parts 폴더를 includes 폴더로 이름 변경 및 경로 정리`
- `feat(login): 회원가입 및 로그인 페이지 마크업 초기 생성`
- `chore(gitignore): 임시 파일 및 빌드 산출물 배제 규칙 추가`
- `docs(log): 2026-06-05 업무일지 작성`
