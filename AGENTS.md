# 🚀 RMHC Korea 퍼블리싱 AI 협업 프로토콜 (AGENTS.md)

본 문서는 RMHC Korea 홈페이지 개편 마크업 프로젝트에 참여하는 모든 AI 에이전트와 퍼블리셔가 준수해야 할 작업 룰, 기술적 아키텍처 및 검증 절차를 기록합니다.

---

## 1. ⚠️ 제1원칙 (The Golden Rules of Operation)

- **전면 한국어 사용**: 모든 질문, 설명, 계획, 보고는 100% 한국어를 사용합니다.
- **추측 절대 금지 (No Guessing / Zero-Hallucination)**:
  - 구현 계획(Implementation Plan)을 작성하기 전에 반드시 `grep_search`, `view_file` 도구를 사용하여 관련 마크업 파일이나 스타일 파일의 존재 여부와 코드를 직접 열고 확인하십시오.
- **의도 임의 판단 금지 (Ask, Don't Assume)**:
  - 사용자의 지시가 함축적이거나 다음 목표가 모호할 때 다음 스텝을 임의로 유추하여 코드를 건드리지 마십시오. 항상 사용자에게 명시적으로 확인 문의를 하십시오.
- **임시 파일 생성 및 정리**:
  - 임시 파일이나 스크립트가 필요한 경우 반드시 프로젝트 최상위의 `scratch/` 디렉토리 하위에서만 생성해야 하며, 목적을 완수한 직후 또는 세션 종료 전에 스스로 삭제하여 작업 공간을 항상 청결하게 유지하십시오.
  - **예외 (피그마 API 덤프 스크립트)**: `scratch/fetch_figma.js` 와 `scratch/parse_figma.js` 는 디자인 폰트 실측 데이터를 피그마 API로 직접 대조 추출하기 위해 삭제하지 않고 유지하며 적극적으로 반복 사용합니다.
- **피그마 API 실측 최우선화 지침**:
  - 앞으로 작업하는 모든 페이지는 피그마 디자인 폰트 사양 검사 시 추측을 절대 배제하고, `scratch/fetch_figma.js` 에 해당 페이지의 피그마 Node ID를 매핑 실행하여 얻어낸 정밀 스타일 명세 수치(font-size, weight 등)를 최우선 기준으로 삼아 코딩을 이행해야 합니다.
- **요청되지 않은 JS 및 모션 작업 금지**:
  - HTML/CSS 작성 시 탭(Tab) 또는 UI 레이아웃 제어를 위해 반드시 필요한 최소한의 스크립트 외에, 임의적인 스크립트 및 모션/애니메이션 작업은 일절 금지합니다.
  - 모든 HTML 및 CSS 마크업 퍼블리싱 작업이 완전히 완료된 후에 별도의 인터랙션 고도화 작업을 진행합니다.
- **적정 커밋 단위 및 task.md 활용**:
  - 너무 작고 미세한 변경(예: 단일 주석 수정, 오타 수정 등)은 즉시 단독으로 커밋하지 않습니다. 이러한 미세 수정은 프로젝트 최상위의 [task.md](file:///d:/rmhc/front/task.md)에 미세 작업으로 기록해 두고, 수정 개수가 최소 5개 이상 누적되었을 때 하나의 논리적 커밋으로 묶어서 진행합니다.
- **개발팀 인도 목적의 마크업 품질 준수**:
  - 이 프로젝트의 최종 산출물은 개발팀에 인도되어 JSP/Thymeleaf/React 등의 템플릿 엔진으로 컨버팅됩니다. 따라서 마크업이 구조적이고 깔끔하며 주석이 체계적으로 작성되어 있어야 합니다.

## 2. 🏗️ 기술 아키텍처 및 폴더 규칙

- **루트 디렉토리 (HTML Entrypoints)**:
  - 기획 IA의 경로에 따라 루트와 하위 폴더(예: `/donate`, `/mypage`)에 각각의 독립적인 HTML 파일들이 위치합니다.
- **공통 영역 인클루드 (`/includes`)**:
  - 헤더, 푸터, 레이아웃 조각, 팝업(`popups/`), 작은 공통 UI 컴포넌트(`components/`)는 `/includes/` 내부에 마크업 조각 파일로 보관합니다.
  - HTML 파일 내에서 `<load ="/includes/header.html" />` 와 같은 플러그인 문법을 통해 상대 경로 깨짐 없이 안전하게 로드합니다. (Vite 번들링 시 정적으로 합쳐집니다.)
- **글로벌 공통 스타일 및 컴포넌트 분리 (`src/css/style.css` 및 `src/css/components/`)**:
  - 이 파일은 공통 스타일 파일입니다. 개별 페이지 전용 스타일 작업은 절대 이곳에 하지 않습니다.
  - 모든 페이지에 공통으로 사용되는 디자인 토큰(색상, 폰트, 반응형 브레이크포인트), 리셋, 그리드 시스템만 이 파일에 선언합니다.
  - 버튼, 폼 등 재사용성이 높고 구조가 큰 공통 UI 요소들은 `src/css/components/` 디렉토리에 개별 컴포넌트 CSS 파일로 작성하고(예: `button.css`, `form.css`), `style.css` 상단에서 `@import`하여 사용함으로써 모듈화를 구현합니다.
- **페이지 개별 스타일 (1:1 매칭 CSS 파일)**:
  - 각 페이지의 전용 스타일은 해당 페이지명과 1:1로 매칭되는 독립적인 CSS 파일로 작업합니다. (예: `index.html` -> `index.css`, `about.html` -> `about.css`)

## 3. 📋 통합 마크업 워크플로우 (Markup Workflow)

```
┌─────────────────────────────────────────────────────┐
│  STEP 1. 팩트 수집 및 플랜 작성                       │
│  ├─ 도구(grep, view)로 파일/코드 팩트 확인            │
│  ├─ implementation_plan.md 작성                     │
│  └─ ⏸️ 사용자 승인 대기                               │
├─────────────────────────────────────────────────────┤
│  STEP 2. 마크업 구현 및 자가 검증                     │
│  ├─ 승인된 계획대로 HTML/CSS/JS 작성                 │
│  ├─ npm run lint:html 실행을 통한 HTML 유효성 검증   │
│  ├─ npm run build 실행을 통한 번들 컴파일 검증        │
│  └─ Lighthouse/PageSpeed 자가 검증 항목 체크          │
├─────────────────────────────────────────────────────┤
│  STEP 3. 업무일지 작성 및 마이크로 커밋               │
│  ├─ docs/업무일지/ 에 일지 작성                       │
│  ├─ git_commit_rules.md 준수하여 커밋 수행            │
│  └─ 결과 보고                                        │
└─────────────────────────────────────────────────────┘
```

## 4. ⚡ PageSpeed Insights & 접근성 검증 체크리스트

모든 페이지 퍼블리싱 시 다음 체크리스트를 머릿속에 담고 준수하십시오.

1. **[성능]** 외부 CDN 폰트 로드 시 `font-display: swap`이 포함되어 있는가?
2. **[접근성]** 모든 버튼 및 클릭 대상 요소의 터치 영역이 최소 `48px` 이상 확보되었는가?
3. **[접근성]** 인풋/라벨 조합이 명확하며, 포커스 시 `:focus-visible` 아웃라인이 또렷하게 드러나는가?
4. **[접근성]** 브랜드 골드/옐로우(`#ffc72c`) 배경 위에 흰색 글씨를 쓰지 않고, 차콜 블랙(`#1f2937`) 글씨를 사용했는가?
5. **[이동]** 모바일 화면에서 Input 입력 시 브라우저가 레이아웃을 억지로 줌인하지 않도록 글꼴 크기를 `16px` 이상으로 정의했는가?
6. **[SEO]** 각 페이지마다 적절한 `<title>`, `<meta name="description">` 태그와 단 하나의 `<h1>` 구조를 갖추었는가?
7. **[개발팀 배려]** 공통으로 쓰이는 리다이렉트나 다이얼로그 팝업(예: `/includes/popups/`) 등은 개발팀이 제어하기 쉽도록 별도 마크업으로 분리되고 가독성 높은 주석이 달려있는가?
8. **[성능]** 페이지 핵심 비주얼(LCP 리소스) 이미지 요소에는 `fetchpriority="high"`가 부여되어 빠르게 로드되는가?
9. **[성능]** 화면 하단이나 비노출 영역의 보조 이미지에는 `loading="lazy"` 속성을 지정하여 초기 로딩 리소스를 절약하는가?
10. **[성능]** 모든 이미지(`<img>`) 태그에 원본 종횡비에 맞는 `width`와 `height` 상수를 명시하여 CLS(누적 레이아웃 이동)를 예방하였는가?
